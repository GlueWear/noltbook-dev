2.8 · ACTOR CONTACTS (Phase F2 · requires %manage-own-contacts)

The current Actor Identity id's own contact book (app:APP + actorObj()). Contacts are tagged references to a ship or another actor — not pals, no trust/membership. This is separate from the host Profile/Contacts/Pals card and never touches /api/contacts.

2.9 · ACTOR PREFERENCES (Phase F3 · requires %manage-own-preferences)

The current Actor Identity id's own stored-only mute/block book (app:APP + actorObj()). Block = hide that identity's content in the plugin; mute = suppress its notifications; they are independent. Stored only — Noltbook does not filter reads or stop posts. Never touches host pal-blocked/contacts/note-muted. This is separate from the host social controls.

2.95 · HOST/USER ACTOR PREFERENCES (Phase B · real ship user, NOT the actor)

The real ship user's notification preferences for ANY actor identity, keyed by the full stable [host, desk, id] (local Rick on ~zod and remote James on ~wet use the same key). Mute = suppress that actor's host green-unread + reply/@~host attention. Block = mute + the frontend hides its message/pin content behind a session VIEW. Neither rejects/deletes the message, suspends/revokes the actor, nor touches host pal-blocked/membership. Notifications are ON by default. This is NOT the actor's own preferences (sec. 2.9), nor app grants, nor actor status/caps. Targets need not exist locally (remote actors valid). Pokes noltbook-api.

8c2 · ACTOR NOTIFICATIONS (Phase G6B · reply-only · requires %manage-own-notifications)

Directed reply notifications for the current Actor Identity (app:APP + Actor Identity id). A reply to a message attributed to this actor creates a row here — NOT host attention. Message-only (mentions/artifact replies deferred); a reply from the same actor to itself is skipped; muted/blocked senders are filtered (the message still posts). To test (both actors active with attribute + post-message + manage-own-notifications, each owning/participating the note): post a root message as Rick with Send (sec. 3); click Rick's message row to select it; switch Actor Identity to Alice; click Reply to Selected Message; switch back to Rick and read here — the row identifies Alice as the replying actor. Live actor-notifications-updated events arrive on /api/results; this read is the durable recovery surface. mark-read and clear are independent — clearing never marks the note read.

8d · ACTOR DM (Phase G5A · requires %send-dm)

A private actor-to-ship direct conversation as a secret two-ship %group note (NOT canonical %dm). Idempotent per [current Actor Identity, target]; Rick→~wet and Alice→~wet get different note IDs. The host @p stays the real member. Adopt lets ONE local actor claim an incoming actor DM (manual actor-to-actor). After create/found the note is selected — use Post Text Message (sec. 3) to chat.

8b · APP METADATA (set-note-app)

Durable note-level association with a Noltbook-capable app (not a note type). Creator/admin only; notebook/group/gossip notes. Reads appear as app in /api/notes and /api/notes/<id>/meta. app:null clears.

8c · ACTIVE (set-note-active)

Developer/API-only note "live" status (separate from calls). Requires top-level app attribution (harness sends APP). Creator/host only; notebook/group/gossip notes. TTL heartbeat in seconds (default 120, cap 600); label caps at 32 (empty → "live"); count caps at 999. Reads appear as active in /api/notes and /api/notes/<id>/meta.

8e · APP NOTIFICATIONS

Public API for app-owned high-level Noltbook/Grimoire notifications. Requires top-level app:APP. Rows are keyed by [desk,id]; TTL is optional, and reads come from /api/app-notifications.

9 · ARTIFACT CREATE

Poke create-artifact (code/app only — no raw file upload in 12A) into the selected note, with APP attribution. Result shows artifactId/eid; the new artifact appears in Read Recent's artifact rows.

10 · SEARCH MESSAGES

Poke search-messages (requestId required) — result arrives on /api/results as api-search-result. Click a hit to open its note. Skips cover/ars-rumors/blocked; cap 50.

11 · DM FIND / CREATE

Poke find-or-create-dm for a target ship. On found/dm-created the DM note is selected + read (idempotent — run twice to get the same noteId). requestId required.

12 · GOSSIP NOTE FIND / CREATE

Poke find-or-create-gossip-note (idempotent by exact name + creator=our + type=gossip). On found/gossip-created the note is selected + read. Creates **user-gossip** notes only.

13 · FORKS

fork-note the selected note (group notes only; member/removed). /api/fork-invites lists pending invites with accept/decline. accept is **async** — it requests a remote fetch + installs later.

14 · CALLS

Noltbook call state only — no media/WebRTC. Start/join/leave operate on the selected note's call. Reads: /api/calls, /api/notes/<id>/calls.
